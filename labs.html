The Labs

Each chapter of the EcoTrack story has a 30–60 minute lab you do on a local Ubuntu VM. The lab gives you a script that creates the broken state from the chapter — your job is to fix it.

👋 New to all this? Read this first.

If you've never used Linux, an SSH session, or a virtual machine — that's fine. These labs assume zero. Every command is explained. Every technical term has a "What is this?" box you can click to read a plain-English definition. Every exercise step has a hint you can reveal if you're stuck for more than 10 minutes.

How to use this page:

  • Read the Concept primer box at the top of each lab — it defines the 4–6 terms used.
  • Hit the "Set the scene" script first; it creates the broken state for the chapter.
  • Work through the exercises one at a time. Don't skip ahead.
  • If a step makes no sense, click "💭 Show a hint". Hints don't give the solution, they nudge you toward one.
  • If a term confuses you, click "💡 What is X?" for a one-paragraph definition.

The 10-minute rule: struggle alone for 10 minutes before opening a hint or asking AI. The struggle is the lesson — productive frustration is how the muscle memory builds.

1. Install Multipass

Multipass is a free virtual machine tool from Canonical (the company that makes Ubuntu). It runs a real Ubuntu Linux inside your normal computer, in a window/process you can start and stop. No company policy or license is required — it's open source.

💡 What is a virtual machine (VM)?

A VM is a real computer simulated inside your computer. It has its own operating system, its own disk, its own memory — but it all runs as a process on your real machine (called the host).

Why use one? Because it's safe. You can install anything, break anything, run weird commands — and your real laptop is untouched. Delete the VM and it all goes away.

# Homebrew (recommended — Homebrew is a package manager for macOS)
brew install --cask multipass

# Or download the .pkg from https://multipass.run/download/macos
multipass version

Works natively on Apple Silicon (M1/M2/M3) — uses QEMU under the hood, no Rosetta needed.

💡 What's Homebrew? What's a package manager?

A package manager is a tool that installs other tools for you. Instead of finding a download page, downloading a file, double-clicking, dragging to Applications, etc. — you type brew install <name> and it handles everything, including updates.

Homebrew is the most popular package manager for macOS. If you don't have it, install it first: brew.sh has a one-line command.

# Snap (Ubuntu, Debian, Fedora, Arch via snapd)
sudo snap install multipass

multipass version
💡 What is sudo?

sudo means "do this command as the administrator". You'll be prompted for your password. It's how Linux protects the system — regular users can't install software or edit system files unless they explicitly ask permission.

Rule of thumb: only use sudo when a command refuses to run without it.

# Download from https://multipass.run/download/windows
# Run the installer. It sets up Hyper-V or VirtualBox automatically.

multipass version
Note: on Windows Home, Multipass uses VirtualBox (it'll install VirtualBox for you). On Pro/Enterprise it uses Hyper-V (which needs to be enabled).

2. Launch the EcoTrack VM

You'll create a single Ubuntu 22.04 VM and reuse it for all 10 labs. 4GB of memory and 20GB of disk is enough for everything, including the Kubernetes labs (we use k3s — a lightweight version that fits in this footprint).

# Launch a new VM named "ecotrack" running Ubuntu 22.04
# Takes ~30 seconds the first time (downloads the image), faster after.
multipass launch --name ecotrack --cpus 2 --memory 4G --disk 20G 22.04

# Check it's running
multipass list

# "Shell into" the VM — you're now inside the Linux machine
multipass shell ecotrack
💡 What does "shell into" mean?

A shell is the program that reads commands you type and runs them. bash and zsh are shells. When you "shell into" a VM, you open a session where every command you type runs inside the VM, not on your laptop.

Your prompt will change — you might see something like ubuntu@ecotrack:~$. That tells you: user ubuntu, on the machine ecotrack, in the home directory (~). The $ is where you type.

To leave the VM and go back to your laptop's shell, type exit or press Ctrl+D.

Now save a clean baseline you can roll back to anytime:

# Snapshots require the VM to be stopped first
multipass stop ecotrack

# Take a snapshot — this is "Save Game" for your VM
multipass snapshot ecotrack --name day-0-fresh --comment "Clean Ubuntu 22.04"

# Start it back up and shell in
multipass start ecotrack
multipass shell ecotrack
💡 What's a snapshot? When should I take one?

A snapshot is a frozen copy of the VM's entire state — disk, configs, installed packages — at a point in time. You can roll back to it later with multipass restore.

Take a snapshot before a lab if you might want to retry from scratch, and after a lab once you've finished it (so you can resume that exact state on Day N+1).

Pretend the VM is prod-1.ecotrack.io. The story addresses you as if you're SSH'd into EcoTrack's production server. Your local Multipass VM is that server. The fiction makes the practice real.

Lab1

"The server keeps crashing"

Linux & Shell · diagnose a leaky service, free a stuffed disk, and set up an early warning

Beginner 45 min
📚 Concept primer — read me first (~2 min)
processA running program. Each one has a number called a PID (process ID).
systemdUbuntu's service manager. It starts programs at boot, restarts them if they crash, and keeps their logs.
service / unitA program managed by systemd. Defined by a .service file. You start/stop/check it with systemctl.
journalctlThe command to read systemd's log database. Like cat, but for service logs across the whole system.
memory leakA bug where a program keeps asking for memory but never gives it back. Eventually it eats all RAM and the OS kills it.
/var/logThe standard Linux folder where applications write log files. When it fills up, things break in weird ways.
cronA built-in scheduler. You give it a command and a schedule (every 5 min, every Monday 3am, etc.) and it runs the command for you.

1Set the scene

Inside the VM (run multipass shell ecotrack if you're not already in), copy the whole block below and paste it into the shell. Press Enter to run it. It installs a leaky service and stuffs the disk — simulating the broken EcoTrack production server.

# --- Day 1 setup: simulate prod-1.ecotrack.io in distress ---

# 1. Create a tiny Python program that leaks memory on purpose
sudo tee /usr/local/bin/ecotrack-api.py > /dev/null <<'PY'
#!/usr/bin/env python3
import time
buf = []
while True:
    buf.append("x" * 10_000_000)  # grab another 10MB of RAM forever
    time.sleep(1)
PY
sudo chmod +x /usr/local/bin/ecotrack-api.py

# 2. Define it as a systemd service, so the OS auto-restarts it on crash
sudo tee /etc/systemd/system/ecotrack-api.service > /dev/null <<'UNIT'
[Unit]
Description=EcoTrack API (simulated, leaky)
After=network.target

[Service]
ExecStart=/usr/bin/python3 /usr/local/bin/ecotrack-api.py
Restart=always
RestartSec=5
MemoryMax=400M

[Install]
WantedBy=multi-user.target
UNIT

# 3. Pile up large dummy log files until the disk is >90% full
sudo mkdir -p /var/log/ecotrack
while :; do
  USED=$(df / | awk 'NR==2 {sub("%","",$5); print $5}')
  [ "$USED" -ge 92 ] && break
  sudo dd if=/dev/zero of=/var/log/ecotrack/old-$RANDOM.log bs=1M count=300 status=none || break
done

# 4. Reload systemd's config and start the leaky service
sudo systemctl daemon-reload
sudo systemctl enable --now ecotrack-api

echo
echo "✗ prod-1.ecotrack.io is unhealthy. Good luck."
💡 What does the script actually do?

Step 1 writes a tiny Python file to /usr/local/bin/ecotrack-api.py. The program is dumb on purpose: it adds 10MB of junk to a list every second forever, never letting go. That's the leak.

Step 2 tells systemd: "treat this Python file as a service called ecotrack-api; if it dies, restart it after 5 seconds; cap its memory at 400MB". The cap means the kernel will kill it long before your VM runs out of RAM — which is exactly the cycle you'll see in the logs.

Step 3 creates large empty files in /var/log/ecotrack/ until the disk is >92% full. Real-world cause: app-rotated logs that nobody cleaned up.

Step 4 tells systemd to reload its config (daemon-reload) and start the service now plus on every boot (enable --now).

2Exercises (~40 min)

1
Get a snapshot of system health.

Run these one at a time. Don't worry if you don't understand every line — read the explanations below first.

uptime    # how long the system has been up + 3 load-average numbers
df -h     # disk usage per filesystem, in human-readable form
free -h   # memory usage, human-readable
top       # live view of all processes (press q to quit)
💡 What do those load-average numbers mean?

Three numbers — averaged over the last 1, 5, and 15 minutes. Each one is "how many processes were waiting for CPU on average". If your VM has 2 CPUs, then 2.0 means CPU is fully used. Anything well above 2.0 means processes are queued up waiting.

So a quick sanity check: load average / number of CPUs. Above ~1.0 = busy. Above ~2.0 = saturated.

💭 Hint — what should I notice?

Look at the % in df -h for the / row. Look at the "available" column in free -h. In top, the top rows are sorted by CPU; press Shift+M (uppercase M) to sort by memory instead. What does the top entry look like?

2
Find the leaking process by name and PID.

Sort all processes by memory usage and look at the worst offender:

ps aux --sort=-%mem | head
💡 What's ps aux? What's a PID?

ps shows running processes. aux is a common combo of flags meaning "show every process owned by every user, in BSD-style output". --sort=-%mem sorts by memory used, descending.

A PID is a process ID — a number Linux assigns to each running process. It's how you refer to a specific running instance, even if many copies of the same program exist.

💭 Hint — what am I looking for?

The first line is column headers: USER PID %CPU %MEM ... COMMAND. The second line is the worst-offending process. The PID is the second column; the program name is at the very end of the row.

3
Read the crash log to confirm the leak.

The leaky service is named ecotrack-api. Read its recent log entries:

journalctl -u ecotrack-api -n 100 --no-pager
💡 What do those flags mean?

-u ecotrack-api — only show logs for the ecotrack-api unit (service).
-n 100 — show the last 100 lines.
--no-pager — print to the terminal directly instead of opening a pager (a pager is a "press space to scroll" view; q to quit).

Other useful variants: journalctl -u ecotrack-api -f tails the log live (Ctrl+C to stop). journalctl -u ecotrack-api --since "10 minutes ago" filters by time.

💭 Hint — what should I see?

Look for repeating lines like "Service died with signal 9" or "exited with status 137". Signal 9 / status 137 means the kernel's OOM killer (Out-Of-Memory) killed the process because it exceeded its memory limit. Then systemd restarts it, it leaks again, gets killed again. That cycle is what you're seeing.

4
Find what's eating the disk and clean it up.

Find the biggest folders in /var/log:

sudo du -sh /var/log/* 2>/dev/null | sort -rh | head -15

Then remove the EcoTrack-rotated logs (only those — don't touch system logs).

💡 What's du doing? Why all those flags?

du = "disk usage".
-s = summarize (show only the total, not every subdirectory).
-h = human-readable (M, G instead of bytes).
2>/dev/null = silence error messages (some folders need root; we ignore those errors).
sort -rh = sort reversed (biggest first), human-aware (knows that 2G > 500M).
head -15 = show only the top 15 lines.

💭 Hint — how do I delete the right files?

You created the mess in /var/log/ecotrack/. Only that folder. Use sudo rm /var/log/ecotrack/*.log — that removes the .log files inside that folder and nothing else.

Never run rm -rf against /var/log or any path you're not 100% sure of. That's how people break production. List with ls first; delete second.

⚠️ Tip: get in the habit of running ls <path> before any rm. The 2 seconds it costs you have saved careers.
5
Stop the bleeding for now.

The leak itself is a code bug — beyond Day 1's scope. Stop the service so the loop ends:

sudo systemctl stop ecotrack-api
💡 stop vs disable vs mask — what's the difference?

stop = stop it right now. It can still start at boot or manually.
disable = don't start it on next boot (but leaves it stoppable/startable).
mask = forbid it entirely. Even root can't start it without unmasking. Use this last one for "this should never run again, ever".

For today, stop is enough. We want it stopped right now.

6
Write a memory watcher script.

Create ~/watch-memory.sh that appends a warning line to /var/log/ecotrack-memory.log if memory usage is over 85%. Make it executable.

💡 How do I create and edit a file in the shell?

Easiest editors on Ubuntu:

  • nano ~/watch-memory.sh — beginner-friendly. Save with Ctrl+O then Enter, exit with Ctrl+X.
  • vim ~/watch-memory.sh — powerful but has a learning curve. Press i to start typing, Esc then :wq to save and quit.

To make a script executable: chmod +x ~/watch-memory.sh. To run it: ~/watch-memory.sh or bash ~/watch-memory.sh.

💭 Hint — how do I get memory usage as a number?

free shows memory; you can pipe its output into awk to extract a percentage:

free | awk '/Mem:/ {printf "%.0f", $3/$2*100}'

That gives a single number like 23. Compare it with -gt 85 in an if:

USE=$(free | awk '/Mem:/ {printf "%.0f", $3/$2*100}')
if [ "$USE" -gt 85 ]; then
    echo "$(date) WARN memory at ${USE}%" | sudo tee -a /var/log/ecotrack-memory.log
fi

Save that as ~/watch-memory.sh with #!/bin/bash as the first line.

7
Schedule it to run every 5 minutes via cron.
crontab -e   # opens your personal cron schedule in an editor

Add this line at the bottom, save, and quit:

*/5 * * * * /home/ubuntu/watch-memory.sh
💡 How do I read cron schedule syntax?

Five fields, space-separated: minute hour day-of-month month day-of-week command.

  • * = "any"
  • */5 = "every 5"
  • 0 3 * * 1 = "at 03:00 every Monday"
  • */5 * * * * = "every 5 minutes, every hour, every day"

Try crontab.guru — paste any cron expression and it tells you in English what it means.

💭 Hint — how do I check it actually ran?

Wait 5 minutes, then check: sudo cat /var/log/syslog | grep CRON | tail. You should see lines for watch-memory.sh. If memory is below 85%, the log file may be empty — that's correct (no warning needed). To force-trigger a write, run the script directly: bash ~/watch-memory.sh.

✅ Verify — you're done when
  • df -h / shows below 70% used
  • systemctl is-active ecotrack-api returns inactive
  • crontab -l shows your watch-memory entry
  • bash ~/watch-memory.sh runs without error (the log file is created if needed)

exit  # leave the VM

Lab2

"A researcher found our open database"

Networking · close the door, audit ports, rotate the password

Beginner 45 min
📚 Concept primer — read me first (~2 min)
portA numbered "door" on a network address. SSH uses 22, HTTPS uses 443, Postgres uses 5432. Programs listen on ports.
listen addressWhich network interface a program accepts traffic on. 127.0.0.1 = only this machine. 0.0.0.0 = anyone, anywhere.
localhost / 127.0.0.1The machine itself. A program listening on 127.0.0.1 can only be reached by other programs on the same machine.
firewallA traffic gate. Decides which incoming packets to allow or drop, regardless of what programs are listening. ufw is Ubuntu's friendly front-end.
PostgreSQL / postgresA popular open-source database. Listens on port 5432 by default. Should never be exposed to the internet.
pg_hba.confPostgres' "who can connect from where" rules. Separate from postgresql.conf (which controls bind, ports, etc.).
nc (netcat)A Swiss Army knife for networking. nc -zv host port = "is something listening on this port?"
ss"Socket statistics" — lists open network connections and listeners. Modern replacement for netstat.

1Set the scene

Make sure the VM is running and you're shelled in: multipass start ecotrack then multipass shell ecotrack. Then paste:

# --- Day 2 setup: PostgreSQL exposed to the world, firewall off ---
sudo apt-get update -qq
sudo DEBIAN_FRONTEND=noninteractive apt-get install -y -qq postgresql ufw netcat-openbsd

# Find the active config dir (varies by Postgres version, e.g. /etc/postgresql/14/main)
PG_DIR=$(sudo find /etc/postgresql -maxdepth 3 -name postgresql.conf | head -1 | xargs dirname)

# Bind to all interfaces (the WRONG thing — open to the world)
sudo sed -i "s/^#\?listen_addresses.*/listen_addresses = '*'/" "$PG_DIR/postgresql.conf"

# Allow password auth from anywhere (the VERY wrong thing)
echo "host all all 0.0.0.0/0 md5" | sudo tee -a "$PG_DIR/pg_hba.conf" > /dev/null

# Set a weak password the "researcher" already guessed
sudo -u postgres psql -c "ALTER USER postgres PASSWORD 'ecotrack123';" >/dev/null

sudo systemctl restart postgresql
sudo ufw --force disable

echo
echo "✗ postgres is listening on $(ss -tlnp | awk '/5432/{print $4}'). UFW is OFF."
💡 What did the script just configure?

Postgres was installed and then deliberately misconfigured:

  • listen_addresses = '*' in postgresql.conf means "accept connections on every network interface" — including the public-facing one.
  • The line in pg_hba.conf says "any host (0.0.0.0/0) can authenticate as any user with a password". That's the "front door wide open" part.
  • The password is ecotrack123. Trivially crackable.
  • ufw --force disable turns off the firewall — there's no second line of defense.

Real-world: this exact configuration drift has leaked millions of records. Your job today: make the door unfindable, then add a deadbolt.

2Exercises (~40 min)

1
Verify the problem yourself, from outside the VM.

From your laptop's shell (not inside the VM), find the VM's IP and probe its Postgres port:

multipass info ecotrack | grep IPv4
nc -zv <that-ip> 5432
💡 What does nc -zv do exactly?

nc (netcat) tries to make a network connection. -z means "don't send any data, just check if the port is open". -v means "verbose — tell me what happened".

If something is listening: Connection to 192.168.x.x 5432 port [tcp/postgresql] succeeded! If not: Connection refused or a timeout.

💭 Hint — what's the security risk?

The connection succeeds, meaning any machine that can reach yours could try to log in. Combined with the weak ecotrack123 password, an attacker could brute-force in within seconds. Once in, they have full access to every database table.

2
Audit every listening port on the VM.

Inside the VM, run:

sudo ss -tlnp
💡 What do those flags mean? How do I read the output?

-t = TCP only (skip UDP). -l = listening sockets only. -n = numeric (don't try to translate ports to names). -p = show the program owning the socket.

The Local Address:Port column tells you what's reachable. If it says 0.0.0.0:5432 or *:5432, the port is open to the world. If it says 127.0.0.1:5432, only the VM itself can reach it.

💭 Hint — for each listener, ask yourself

"Should the public internet be able to talk to this?" SSH on port 22 — yes (you need to log in). HTTPS on 443 — yes if you serve traffic. Postgres on 5432 — no. Anything else binding to 0.0.0.0 or ::: — investigate.

3
Lock Postgres to localhost only.

Two files to fix:

  • /etc/postgresql/<version>/main/postgresql.conf — change listen_addresses = '*' to listen_addresses = 'localhost'
  • /etc/postgresql/<version>/main/pg_hba.conf — remove the 0.0.0.0/0 line; allow only 127.0.0.1/32 (and ::1/128 for IPv6)

Then restart Postgres and re-test:

sudo systemctl restart postgresql
sudo ss -tlnp | grep 5432         # should show only 127.0.0.1:5432 now
sudo -u postgres psql -c '\l'     # local connection still works

From your laptop again: nc -zv <vm-ip> 5432 — should now fail.

💡 How do I edit those config files?

They're in /etc/postgresql/... — root-owned. Use sudo nano (or sudo vim):

sudo nano /etc/postgresql/14/main/postgresql.conf
sudo nano /etc/postgresql/14/main/pg_hba.conf

The version number (e.g. 14) might differ — use Tab to autocomplete or ls /etc/postgresql/ to check.

💭 Hint — exact text to search for

In postgresql.conf, find the line starting with listen_addresses. There may be a # comment in front — remove it. Set the value to 'localhost'.

In pg_hba.conf, scroll to the bottom. Delete (or comment out with #) the line we appended: host all all 0.0.0.0/0 md5. The default 127.0.0.1/32 entries that came with the package install are fine.

4
Configure the firewall: deny by default, allow only SSH.
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw allow 22/tcp
sudo ufw --force enable
sudo ufw status verbose
💡 Why both "Postgres bound to localhost" AND "firewall blocks 5432"?

Defense in depth. Two layers, each independent:

  • If a future config change accidentally re-binds Postgres to 0.0.0.0, the firewall still blocks it.
  • If the firewall is misconfigured, the bind setting still keeps it private.

One layer alone is fragile. Both layers together is robust.

💭 Hint — what about "I'm SSH'd in, will this kick me out?"

You added the SSH allow rule (ufw allow 22/tcp) before enabling the firewall, so existing SSH sessions stay alive. Multipass also uses its own networking, so even without that rule the multipass shell command would still work. But on a real cloud server: always allow port 22 BEFORE enabling ufw, or you'll lock yourself out.

5
Rotate the postgres password.

Generate something strong, set it in Postgres, store it for your app to read — never put it in shell history.

💭 Hint — full sequence

Generate a strong random password and use a here-string so it doesn't end up in your ~/.bash_history:

NEW_PW=$(openssl rand -base64 24)
sudo -u postgres psql -c "ALTER USER postgres PASSWORD '$NEW_PW';"

# Save it to ~/.pgpass (Postgres' standard credentials file)
echo "localhost:5432:*:postgres:$NEW_PW" > ~/.pgpass
chmod 600 ~/.pgpass
unset NEW_PW   # clear it from this shell's environment

chmod 600 = "owner can read/write, nobody else can even read". Postgres refuses to use a .pgpass file with looser permissions — that's the protection.

⚠️ Tip: never type a password literally on a command line in production. It ends up in your shell history file, in process listings, and in audit logs. Use environment variables, password managers, or files with strict permissions.
6
Conceptual checkpoint — write or draw.

In your own words (notes app, paper, anywhere — but write it), describe what happens network-wise when a packet from the public internet hits port 5432 on this VM:

  • Before your fixes
  • After your fixes

If you can explain both clearly, you've internalized the lesson. If you can't, that's fine — re-read the primer and try again. The act of writing it out is what builds the understanding.

✅ Verify — you're done when
  • sudo ss -tln | grep 5432 shows only 127.0.0.1:5432 (no 0.0.0.0 or :::)
  • sudo ufw status shows active, with port 22 allowed and a default deny policy
  • From the host: nc -zv <vm-ip> 5432 fails (timeout or refused)
  • From the host: nc -zv <vm-ip> 22 succeeds
  • ls -l ~/.pgpass shows mode -rw-------
