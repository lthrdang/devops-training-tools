The LabsBài thực hành

Each chapter of the EcoTrack story has a 30–60 minute lab you do on a local Ubuntu VM. The lab gives you a script that creates the broken state from the chapter — your job is to fix it. Mỗi chương của câu chuyện EcoTrack có một bài thực hành 30–60 phút trên máy ảo Ubuntu cục bộ. Mỗi bài cung cấp script tạo ra tình trạng lỗi từ chương đó — nhiệm vụ của bạn là sửa nó.

👋 New to all this? Read this first.Mới bắt đầu? Đọc phần này trước.

If you've never used Linux, an SSH session, or a virtual machine — that's fine. These labs assume zero. Every command is explained. Every technical term has a "What is this?" box you can click to read a plain-English definition. Every exercise step has a hint you can reveal if you're stuck for more than 10 minutes. Nếu bạn chưa từng dùng Linux, SSH hay máy ảo — không sao cả. Các bài này giả định bạn chưa biết gì. Mọi lệnh đều được giải thích. Mọi thuật ngữ kỹ thuật đều có ô "Cái này là gì?" để bạn nhấn vào đọc giải thích bằng ngôn ngữ đơn giản. Mỗi bước bài tập đều có gợi ý nếu bạn bị tắc hơn 10 phút.

1. Install MultipassCài đặt Multipass

Multipass is a free virtual machine tool from Canonical (the company that makes Ubuntu). It runs a real Ubuntu Linux inside your normal computer. No company policy or license required — it's open source. Multipass là công cụ máy ảo miễn phí từ Canonical (công ty tạo ra Ubuntu). Nó chạy Ubuntu Linux thực sự bên trong máy tính bình thường của bạn. Không cần chính sách công ty hay giấy phép — nó là mã nguồn mở.

💡 What is a virtual machine (VM)?Máy ảo (VM) là gì?

A VM is a real computer simulated inside your computer. It has its own operating system, disk, and memory — but it all runs as a process on your real machine (called the host). Why use one? You can install anything, break anything, and your real laptop is untouched. Delete the VM and it all goes away.Máy ảo là một máy tính thực sự được mô phỏng bên trong máy tính của bạn. Nó có hệ điều hành, ổ đĩa và bộ nhớ riêng — nhưng tất cả chạy như một tiến trình trên máy thực của bạn (gọi là host). Tại sao dùng máy ảo? Bạn có thể cài bất cứ thứ gì, phá bất cứ thứ gì, còn laptop thực của bạn không bị ảnh hưởng.

# Homebrew (recommended)
brew install --cask multipass

multipass version

Works natively on Apple Silicon (M1/M2/M3).Chạy tốt trên Apple Silicon (M1/M2/M3).

💡 What's Homebrew?Homebrew là gì?

A package manager for macOS. Instead of downloading and installing manually, you type brew install <name>. If you don't have it, install it from brew.sh.Trình quản lý gói cho macOS. Thay vì tải và cài thủ công, bạn gõ brew install <tên>. Nếu chưa có, cài từ brew.sh.

# Snap
sudo snap install multipass
multipass version
💡 What is sudo?sudo là gì?

sudo means "run this as administrator". Linux protects system files from regular users — sudo is how you explicitly ask for elevated permission. Rule: only use it when a command refuses to run without it.sudo nghĩa là "chạy lệnh này với quyền quản trị viên". Linux bảo vệ file hệ thống khỏi người dùng thông thường — sudo là cách bạn xin quyền nâng cao. Quy tắc: chỉ dùng khi lệnh từ chối chạy mà không có nó.

# Download from https://multipass.run/download/windows and run the installer
multipass version
Note:Lưu ý: On Windows Home, Multipass uses VirtualBox (installs it for you). On Pro/Enterprise it uses Hyper-V.Trên Windows Home, Multipass dùng VirtualBox (tự cài cho bạn). Trên Pro/Enterprise thì dùng Hyper-V.

2. Launch the EcoTrack VMKhởi động máy ảo EcoTrack

Create a single Ubuntu 22.04 VM and reuse it for all 10 labs. 4GB memory and 20GB disk is enough for everything, including the Kubernetes labs (we use k3s — lightweight). Tạo một máy ảo Ubuntu 22.04 và dùng lại cho cả 10 bài thực hành. 4GB RAM và 20GB ổ đĩa là đủ cho mọi thứ, kể cả bài Kubernetes (chúng ta dùng k3s — nhẹ hơn nhiều). 

# Tạo máy ảo tên "ecotrack" — lần đầu mất ~30 giây để tải image
multipass launch --name ecotrack --cpus 2 --memory 4G --disk 20G 22.04

# Kiểm tra đang chạy
multipass list

# Vào trong máy ảo (shell vào VM)
multipass shell ecotrack
💡 What does "shell into" mean?"Shell vào" nghĩa là gì?

A shell is the program that reads and runs your commands (bash, zsh). When you "shell into" a VM, every command you type runs inside the VM, not on your laptop. Your prompt changes to something like ubuntu@ecotrack:~$. To leave and go back to your laptop, type exit or press Ctrl+D.Một shell là chương trình đọc và chạy lệnh của bạn (bash, zsh). Khi bạn "shell vào" máy ảo, mọi lệnh bạn gõ đều chạy bên trong máy ảo, không phải trên laptop. Dấu nhắc lệnh thay đổi thành ubuntu@ecotrack:~$. Để thoát về laptop, gõ exit hoặc nhấn Ctrl+D.

Save a clean baseline snapshot — you can always roll back to this:Lưu một snapshot cơ sở sạch — bạn có thể khôi phục lại bất cứ lúc nào:

# Snapshot cần VM tắt trước
multipass stop ecotrack
multipass snapshot ecotrack --name day-0-fresh --comment "Clean Ubuntu 22.04"
multipass start ecotrack
multipass shell ecotrack
💡 What's a snapshot?Snapshot là gì?

A snapshot is a frozen copy of the VM's entire state — disk, configs, installed packages — at a point in time. Roll back with multipass restore. Think of it as "Save Game" for your VM. Take one before each lab and after completing it.Snapshot là bản sao đông lạnh toàn bộ trạng thái máy ảo — ổ đĩa, cấu hình, phần mềm đã cài — tại một thời điểm. Khôi phục bằng multipass restore. Hãy nghĩ đó là "Lưu game" cho máy ảo. Chụp snapshot trước và sau mỗi bài thực hành.

Pretend the VM is prod-1.ecotrack.io.Hãy giả vờ máy ảo là prod-1.ecotrack.io. The story addresses you as if you're SSH'd into EcoTrack's production server. Your local VM is that server. Câu chuyện xưng hô với bạn như thể bạn đang SSH vào máy chủ production của EcoTrack. Máy ảo cục bộ của bạn chính là máy chủ đó.

1Set the sceneDựng cảnh

Inside the VM (multipass shell ecotrack), paste the whole block below and press Enter. It installs a leaky service and stuffs the disk — simulating the broken EcoTrack production server.Trong máy ảo (multipass shell ecotrack), dán toàn bộ đoạn dưới đây và nhấn Enter. Nó cài một dịch vụ bị rò rỉ và lấp đầy ổ đĩa — mô phỏng máy chủ production EcoTrack bị hỏng.

# --- Day 1 setup: simulate prod-1.ecotrack.io in distress ---

# 1. Tạo chương trình Python cố ý rò rỉ bộ nhớ
sudo tee /usr/local/bin/ecotrack-api.py > /dev/null <<'PY'
#!/usr/bin/env python3
import time
buf = []
while True:
    buf.append("x" * 10_000_000)  # chiếm thêm 10MB RAM mỗi giây mãi mãi
    time.sleep(1)
PY
sudo chmod +x /usr/local/bin/ecotrack-api.py

# 2. Định nghĩa nó là một systemd service
sudo tee /etc/systemd/system/ecotrack-api.service > /dev/null <<'UNIT'
[Unit]
Description=EcoTrack API (simulated, leaky)
After=network.target

[Service]
ExecStart=/usr/bin/python3 /usr/local/bin/ecotrack-api.py
Restart=always
RestartSec=5
MemoryMax=400M

[Install]
WantedBy=multi-user.target
UNIT

# 3. Tạo file log giả cho đến khi ổ đĩa >90% đầy
sudo mkdir -p /var/log/ecotrack
while :; do
  USED=$(df / | awk 'NR==2 {sub("%","",$5); print $5}')
  [ "$USED" -ge 92 ] && break
  sudo dd if=/dev/zero of=/var/log/ecotrack/old-$RANDOM.log bs=1M count=300 status=none || break
done

# 4. Tải lại cấu hình systemd và khởi động dịch vụ
sudo systemctl daemon-reload
sudo systemctl enable --now ecotrack-api

echo
echo "✗ prod-1.ecotrack.io đang có vấn đề. Chúc may mắn."
💡 What does the script actually do?Script thực sự làm gì?

Step 1 writes a tiny Python file. The program grabs 10MB of RAM every second and never lets go — that's the leak. Step 2 tells systemd: restart it after 5 seconds if it dies; cap memory at 400MB. The kernel kills it when it hits 400MB, systemd restarts it, it leaks again — that's the crash loop. Step 3 fills /var/log/ecotrack/ with dummy files until disk is >92% full. Step 4 starts it all.Bước 1 viết một file Python nhỏ. Chương trình chiếm 10MB RAM mỗi giây và không bao giờ thả ra — đó là rò rỉ. Bước 2 nói với systemd: khởi động lại sau 5 giây nếu chết; giới hạn bộ nhớ 400MB. Kernel kill nó khi chạm 400MB, systemd khởi động lại, nó rò rỉ lại — đó là vòng lặp crash. Bước 3 lấp đầy /var/log/ecotrack/ bằng file giả cho đến khi ổ đĩa >92% đầy. Bước 4 khởi động tất cả.

2Exercises (~40 min)Bài tập (~40 phút)

1
Get a snapshot of system health.Kiểm tra tình trạng sức khỏe hệ thống.

Run these one at a time and read the output:Chạy từng lệnh một và đọc kết quả:

uptime    # thời gian hoạt động + 3 chỉ số load average
df -h     # dung lượng ổ đĩa, dạng dễ đọc
free -h   # dung lượng bộ nhớ, dạng dễ đọc
top       # xem trực tiếp tiến trình (nhấn q để thoát)
💡 What does load average mean?Load average nghĩa là gì?

Three numbers — averaged over the last 1, 5, and 15 minutes. Each is "how many processes were waiting for CPU". On a 2-CPU VM, anything above 2.0 means CPU is saturated.Ba con số — trung bình trong 1, 5 và 15 phút vừa qua. Mỗi số là "số tiến trình đang chờ CPU". Trên máy ảo 2 CPU, bất kỳ con số nào trên 2.0 có nghĩa là CPU đang quá tải.

💭 Hint — what should I notice?Gợi ý — tôi cần chú ý gì?

Look at the % in df -h for the / row. Check "available" in free -h. In top, press Shift+M to sort by memory. What's at the top?Nhìn vào % trong df -h ở hàng /. Kiểm tra "available" trong free -h. Trong top, nhấn Shift+M để sắp xếp theo bộ nhớ. Cái gì đứng đầu?

2
Find the leaking process by name and PID.Tìm tiến trình rò rỉ theo tên và PID.
ps aux --sort=-%mem | head
💡 What's ps aux? What's a PID?ps aux là gì? PID là gì?

ps shows running processes. aux = show all processes for all users. --sort=-%mem = sort by memory, biggest first. A PID is a process ID — a unique number Linux assigns to each running process.ps hiển thị các tiến trình đang chạy. aux = hiển thị tất cả tiến trình của mọi user. --sort=-%mem = sắp xếp theo bộ nhớ, lớn nhất trước. PID là mã tiến trình — số duy nhất Linux gán cho mỗi tiến trình đang chạy.

💭 Hint — what am I looking for?Gợi ý — tôi đang tìm gì?

The second line (first data row) is the worst offender. The PID is column 2; the program name is at the end.Dòng thứ hai (hàng dữ liệu đầu tiên) là tiến trình tệ nhất. PID là cột 2; tên chương trình ở cuối dòng.

3
Read the crash log to confirm the leak.Đọc log crash để xác nhận rò rỉ.
journalctl -u ecotrack-api -n 100 --no-pager
💡 What do those flags mean?Các flag đó nghĩa là gì?

-u ecotrack-api = only show logs for this service. -n 100 = last 100 lines. --no-pager = print directly to terminal instead of opening a scroll view. Useful variant: -f to follow logs live.-u ecotrack-api = chỉ hiện log cho dịch vụ này. -n 100 = 100 dòng cuối. --no-pager = in thẳng ra terminal thay vì mở chế độ cuộn. Biến thể hữu ích: -f để theo dõi log trực tiếp.

💭 Hint — what should I see?Gợi ý — tôi sẽ thấy gì?

Look for "signal 9" or "status 137" — that means the OOM killer (Out-Of-Memory) terminated the process for exceeding its memory limit. Then systemd restarts it, it leaks again, gets killed again. That cycle is the crash loop.Tìm "signal 9" hoặc "status 137" — nghĩa là OOM killer (tiến trình kill khi hết bộ nhớ) đã kết thúc tiến trình vì vượt giới hạn bộ nhớ. Rồi systemd khởi động lại, nó rò rỉ lại, bị kill lại. Vòng lặp đó là crash loop.

4
Find what's eating the disk and clean it up.Tìm thứ đang ngốn ổ đĩa và dọn sạch.
sudo du -sh /var/log/* 2>/dev/null | sort -rh | head -15
💡 What's du doing?du đang làm gì?

du = "disk usage". -s = summarize (total only). -h = human-readable (M, G). 2>/dev/null = silence permission errors. sort -rh = sort reversed, human-aware. head -15 = show top 15.du = "disk usage" (dung lượng ổ đĩa). -s = tóm tắt (chỉ tổng). -h = dạng dễ đọc (M, G). 2>/dev/null = ẩn lỗi quyền. sort -rh = sắp xếp đảo ngược, nhận biết đơn vị. head -15 = hiện top 15.

💭 Hint — how do I delete the right files?Gợi ý — làm thế nào xóa đúng file?

The mess is in /var/log/ecotrack/ — only that folder. Use sudo rm /var/log/ecotrack/*.log. Never run rm -rf against a path you're not 100% sure of. Always ls <path> first.Rác nằm ở /var/log/ecotrack/ — chỉ thư mục đó. Dùng sudo rm /var/log/ecotrack/*.log. Đừng bao giờ chạy rm -rf với đường dẫn bạn không chắc 100%. Luôn ls <đường dẫn> trước.

⚠️ Tip:Mẹo: Run ls before any rm. The 2 seconds it costs you have saved careers.Chạy ls trước mọi lệnh rm. 2 giây đó đã cứu nhiều sự nghiệp.
5
Stop the service for now.Dừng dịch vụ lại.

The leak is a code bug — beyond Day 1's scope. Stop the service so the loop ends:Rò rỉ là lỗi code — ngoài phạm vi Ngày 1. Dừng dịch vụ để kết thúc vòng lặp:

sudo systemctl stop ecotrack-api
💡 stop vs disable vs mask — what's the difference?stop vs disable vs mask — khác gì nhau?

stop = stop right now, can still start at boot. disable = don't auto-start on next boot. mask = forbid completely — even root can't start it. For today, stop is enough.stop = dừng ngay, vẫn có thể khởi động khi boot. disable = không tự khởi động khi boot tiếp theo. mask = cấm hoàn toàn — kể cả root cũng không khởi động được. Hôm nay, stop là đủ.

6
Write a memory watcher script.Viết script theo dõi bộ nhớ.

Create ~/watch-memory.sh that appends a warning line to /var/log/ecotrack-memory.log when memory > 85%. Make it executable.Tạo ~/watch-memory.sh thêm dòng cảnh báo vào /var/log/ecotrack-memory.log khi bộ nhớ > 85%. Đặt quyền thực thi cho nó.

💡 What does nc -zv do?nc -zv làm gì?

💡 Why both "Postgres bound to localhost" AND "firewall blocks 5432"?Tại sao cần cả hai: "Postgres bind localhost" VÀ "tường lửa chặn 5432"?

If the firewall is misconfigured, the bind setting keeps it private. One layer alone is fragile. Both together is robust.Bảo vệ theo chiều sâu.
