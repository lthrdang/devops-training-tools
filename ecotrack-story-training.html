You just joined EcoTrack.
Now fix it.

One company. One broken system. Ten escalating real-world problems — each one unlocked by solving the previous. Every skill you learn is the solution to the next chapter.

The Company

EcoTrack — B2B SaaS for corporate CO₂ tracking. 4-person team, 80 paying customers, Series A pending.

The Stack

Node.js API · React frontend · PostgreSQL · hosted on AWS & GCP

Your Role

First DevOps engineer. The founders built it fast. Your job is to make it production-grade.

Day 1 · Linux & Shell
"The Server is Sick"
Production keeps crashing. Nobody knows why. You have SSH access and 4 hours.
bash systemd journalctl 🔴 Production issue
Up next ▾
📖 The Situation
It's your first morning at EcoTrack. Before you've even finished setting up your laptop, the CTO Minh sends you a message.
M
Minh (CTO) · 9:03 AM
Hey! Welcome aboard 🎉 Quick heads up — the production server has been "acting weird" for two days. The app crashes every few hours, we just SSH in and restart it manually. I have no idea what's actually wrong. Here are the credentials: ssh deploy@prod-1.ecotrack.io. Can you take a look when you get a chance? (No rush... well, maybe some rush. Two customers emailed already.)
🎯 Your Mission
SSH in. Get a full picture: uptime, disk, CPU, memory. Something is wrong — find what it is before touching anything.
Find the process that's leaking memory. Use top, htop, or ps aux --sort=-%mem. Identify it by name and PID.
Read the crash logs. The app runs as a systemd service named ecotrack-api. Use journalctl to find the actual error that caused the last 3 crashes.
The disk is 94% full. Find what's eating it (du -sh /* 2>/dev/null | sort -rh | head -15) and clean it up safely without deleting anything important.
Write a bash script watch-memory.sh that runs every 5 minutes via cron, and appends a warning line to /var/log/ecotrack-memory.log if memory usage exceeds 85%.
✅ This chapter is done when
You can tell Minh: "The crash was caused by [X]. The disk was full because of [Y], I cleaned it. I've set up a cron job that will warn us before memory becomes critical again." — and you can prove each claim with a command or log output.
🤖 AI Challenge
Paste the actual journalctl output into Claude and use this prompt:
"Here are the crash logs from our Node.js app running on Linux: [paste your journalctl output] Walk me through what each line means. What is the root cause? What would you check next? Don't fix it for me — ask me guiding questions so I can figure it out."
Rule: Claude is your mentor, not your answer machine. You must explain the fix to your trainer in your own words.

Day 2 · Networking
"Our Database is on the Internet"
A security researcher emails: "Your PostgreSQL is publicly reachable. I could read your entire customer database."
ufw netstat / ss curl 🔴 Security incident
Locked ▾
📖 The Situation
Day 2. You fixed the crashes — good. But Minh forwards you an email with a pale face.
M
Minh (CTO) · 8:47 AM
We got an email from someone claiming to be a security researcher. They said port 5432 (Postgres) is open on our server and they accessed it using the default credentials they found in our old GitHub repo. They sent a screenshot of our customers table. I think they're telling the truth. What do we do??
🎯 Your Mission
Verify it yourself first. From your local machine, try nc -zv prod-1.ecotrack.io 5432. Understand what you see and why it's dangerous.
Immediately lock down PostgreSQL: configure it to bind to 127.0.0.1 only (edit postgresql.conf and pg_hba.conf). Restart and verify it's no longer reachable externally.
Audit all open ports on the server with ss -tlnp. The only ports that should be reachable from the internet are 22 (SSH) and 443 (HTTPS). Close everything else with ufw.
Change the database password immediately. Update the app's config. Verify the app still works after the change.
Write a 5-line security incident summary for Minh: what was exposed, for how long (check server access logs), what you did to fix it, and what to do differently in future.
✅ This chapter is done when
From the internet: port 5432 connection refused ✓ · Only ports 22 and 443 reachable ✓ · App still works with the new credentials ✓ · Incident summary written ✓

Ask Claude to be the security auditor:
"Act as a penetration tester. I'm a DevOps engineer who just locked down a Linux server. I've allowed only ports 22 and 443 via ufw, and moved PostgreSQL to localhost-only. What else would you check to make sure this server is properly hardened? Give me a checklist."
Go through the checklist. Try to implement at least 2 more items beyond what was in the mission.

Day 3 · Docker
"It Works on My Laptop"
Sara deploys a new feature. It crashes on the server. The cause: different Node.js versions, a missing native dependency, and zero reproducibility.
docker Dockerfile docker-compose
Locked ▾
📖 The Situation
The server is locked down. You're feeling good. Then Sara, the backend dev, pings you.
S
Sara (Backend) · 2:14 PM
Hey, I deployed the new emissions chart feature. It works perfectly on my machine but the server is throwing: Error: Cannot find module 'canvas' — was compiled against a different Node.js version. I've been stuck for 3 hours. Minh is watching. Help?

You check. The server is running Node 16. Sara's MacBook has Node 20. The canvas native module compiled for her machine doesn't run on the server. This will happen again. And again. The only real fix is to make the environment identical everywhere.
🎯 Your Mission
Write a Dockerfile for the EcoTrack API. Start from node:20-alpine. Copy only what's needed. Install dependencies. Expose the right port. Use a non-root user.
Make it a multi-stage build: a builder stage that compiles native modules, and a lean runtime stage that only has the final app. Compare image sizes before and after.
Write a docker-compose.yml with three services: api, db (postgres:15), and redis. The API should start only after the DB is healthy. Use a named volume for the DB data.
Test it: docker-compose down -v && docker-compose up --build. The app should start cleanly with no manual steps. Verify Sara can run this on her Mac with zero setup.
Add a .dockerignore file. Make sure node_modules, .git, and local .env files are excluded from the image.
✅ This chapter is done when
Sara runs docker-compose up on her MacBook and the app works — without installing Node, Postgres, or Redis manually. The image is under 200MB. There are no root processes. docker-compose down -v cleans up completely.
🤖 AI Challenge
Paste your Dockerfile into Claude and ask:
"Review this Dockerfile for a Node.js API: [paste your Dockerfile] Check for: security issues (running as root?), image size inefficiencies, layer cache problems, and missing best practices. Don't rewrite it — explain what's wrong and why, then let me fix it."
Apply all valid feedback. Then ask Claude: "What would a production-grade Dockerfile add that mine doesn't have yet?"

A journalist wrote about EcoTrack in a climate tech newsletter. It got picked up by TechCrunch. Traffic went from 200 req/min to 8,000 req/min in under 20 minutes. The single Docker container hit its memory limit and died. The app is down. Minh is on a call with two potential Series A investors who are watching the site not load.
M
Minh (CTO) · 11:32 AM
THE APP IS DOWN. TechCrunch article just went live. Investors on a call RIGHT NOW and the site is timing out. Can you fix this?? I know we talked about Kubernetes last week — now is the time.
🎯 Your Mission
Set up a local Kubernetes cluster (minikube or kind) to test against. Push the Docker image from Day 3 to a registry (Docker Hub or local).
Write a Deployment manifest for the EcoTrack API: 3 replicas, correct image, resource requests and limits, environment variables from a ConfigMap.
Expose it with a Service (ClusterIP first, then NodePort so you can test it). Verify all 3 pods are running and traffic is load-balanced across them.

Deploy a v2 of the app (change something visible, e.g. an API response field). Use a rolling update. Watch the rollout with kubectl rollout status. Then roll it back.

3 replicas running. Kill one — app stays up, Kubernetes self-heals. Deploy v2 with zero downtime. Roll back in under 30 seconds. You can explain to Minh what Kubernetes is doing and why this is better than one container.

Day 5 · Helm + K8s Advanced
"Sara Deployed to Prod Again"
No staging environment, configs copy-pasted between envs, one mistake breaks customers. The team needs proper environment management.
Helm Namespaces Secrets Probes
Locked ▾
📖 The Situation
Sara was testing a new API endpoint. She applied her Kubernetes manifests, checked the app, everything looked good. Then Minh gets a support email: "Our production dashboard is showing test data." Sara had applied to the wrong namespace — there effectively wasn't one. Staging and prod were sharing the same cluster config.
S
Sara (Backend) · 4:01 PM
I'm so sorry. I thought I was in staging. Can we please have a proper staging environment? I'm scared to deploy anything right now.
🎯 Your Mission
Add liveness and readiness probes to the Deployment from Day 4. Intentionally misconfigure the liveness probe to see what CrashLoopBackOff looks like — then fix it.
Create a Kubernetes Secret for the database password. Mount it as an environment variable in the Deployment. Verify it never appears in plaintext in your YAML files.
Create a Helm chart for EcoTrack: helm create ecotrack, then replace the boilerplate with your real app templates (Deployment, Service, ConfigMap, Secret reference).
Create two values files: values-staging.yaml (1 replica, staging DB URL) and values-production.yaml (3 replicas, prod DB URL, resource limits). Deploy both to separate namespaces.
Simulate Sara's mistake: try to apply the staging chart to the production namespace. Verify it's impossible to mix them up now. Update the team wiki with: "how to deploy to staging" in 3 commands.
✅ This chapter is done when
Two environments, one chart. helm upgrade --install ecotrack-staging ./ecotrack -f values-staging.yaml -n staging and helm upgrade --install ecotrack-prod ./ecotrack -f values-production.yaml -n production both work cleanly. Sara can deploy to staging without ever touching production.

After generating, explain what each template file does and what would break if I deleted it."

Day 6 · Cloud Foundations
"The Investors Want a Security Audit"
Series A term sheet requires SOC2 readiness. Auditors find: no VPC isolation, hardcoded IAM credentials, database reachable from the internet.
AWS / GCP VPC IAM ⚠️ Compliance
Locked ▾
📖 The Situation
EcoTrack just signed a Series A term sheet. The investor's technical due diligence team spent two hours on your infrastructure. Their report:
M
Minh (CTO) · 9:15 AM
Okay so the audit came back. The findings: (1) no VPC — everything is on default networking, (2) an IAM user with AdministratorAccess and hardcoded keys in the codebase, (3) RDS instance in a public subnet, (4) S3 bucket with public read. They want these fixed before close. We have 2 weeks.
🎯 Your Mission
Create a VPC with 2 public subnets and 2 private subnets across 2 availability zones. Add a NAT Gateway so private instances can reach the internet but can't be reached from it.
Create IAM roles (not users) for each component: an EC2/EKS role with only S3 read access, an RDS access role, and a CI/CD deploy role. No hardcoded keys anywhere.
Move the database into the private subnet. Verify it's unreachable from your laptop but reachable from an EC2 in the private subnet.
Create an S3 bucket for customer emissions reports. Enable versioning, Block Public Access, and a lifecycle policy that moves objects older than 90 days to cheaper storage.
Use only the CLI (no console clicks) for everything. Document each command in a setup.sh so it could be reproduced.
✅ This chapter is done when
Security checklist passes: DB unreachable from internet ✓ · No credentials hardcoded anywhere in code ✓ · S3 bucket not public ✓ · Each service has its own least-privilege IAM role ✓ · Everything was built via CLI, not console ✓

Day 7 · Terraform
"I Accidentally Deleted Staging"

Terraform Remote state Modules
Locked ▾
📖 The Situation
James, a new backend engineer, was cleaning up old resources in the AWS console. He saw "ecotrack-staging" and thought it was an old test environment. He deleted the VPC, the subnets, the security groups, and the RDS instance.
J
James · 3:48 PM
I'm so sorry. I thought those were old test resources. I deleted staging. I've been trying to rebuild it for 4 hours and I can't get the networking right. How long will this take to fix?
You pull up the Terraform files you'd been writing since Day 6. You type terraform apply. Nineteen minutes later, staging is back.
🎯 Your Mission
Write Terraform code that provisions the full EcoTrack infra from Day 6: VPC + subnets + security groups + IAM roles + S3 bucket. It should match exactly what you built manually.
Set up a remote backend: S3 bucket for state storage + DynamoDB table for state locking. Never store state locally in a real project.
Extract the VPC + subnets into a reusable Terraform module. Call it twice — once for staging, once for production — with different CIDR ranges.
Simulate James's incident: manually delete a resource in the console, then run terraform plan. Observe what it detects. Run terraform apply to restore it.
Add an IAM policy that denies anyone except your Terraform role from deleting the VPC and subnets. James can never do this again.
✅ This chapter is done when
Run terraform destroy then terraform apply. The entire staging environment rebuilds in under 25 minutes with zero manual steps. State is remote. Two environments share one module. Infrastructure drift is detectable.

Day 8 · CI/CD
"Nobody Should SSH to Deploy"
Manual deploys have broken production twice. Someone forgot npm install. Another forgot to set an env var. It's time to automate the human out of the process.
GitHub Actions Docker registry Helm deploy
Locked ▾
📖 The Situation
The current deploy process: SSH into the server, git pull, npm install, pm2 restart. Last Friday at 5pm someone forgot npm install after pulling a commit that added a new dependency. The app crashed silently. It took 45 minutes before a customer reported it.
M
Minh (CTO) · Mon 9:00 AM
After Friday's incident — can we please make it so nobody can break production by doing a deploy wrong? I want it to be impossible to deploy something broken.
🎯 Your Mission
Create a GitHub Actions workflow that triggers on push to main: run tests, lint, and build the Docker image. If any step fails, the workflow stops.
On success, push the image to a registry (Docker Hub or ECR) tagged with the git commit SHA. Use GitHub Secrets for credentials — never in the YAML.
Add a deploy step that runs helm upgrade to update the staging namespace automatically. Production requires a manual approval gate before deploying.
Test the failure path: intentionally break a test. Verify the pipeline stops and nothing is deployed. Fix it and verify the pipeline passes and deploys correctly.
Add a Slack notification (or email) that fires on pipeline failure. The on-call engineer must be notified within 2 minutes of a failed deploy.
✅ This chapter is done when
Nobody SSHes to deploy. A broken test blocks the pipeline ✓ · Image is tagged with commit SHA ✓ · Staging deploys automatically, prod requires approval ✓ · Failed deploy triggers a notification ✓ · The team's deploy process is now: merge a PR.
🤖 AI Challenge
"Review this GitHub Actions workflow: [paste your workflow YAML] Check for: hardcoded secrets, missing permissions, no caching (slow builds), race conditions if two deploys run simultaneously, and anything that would make this unsuitable for production. Give me specific line-level feedback."
Then ask: "How would you add a canary deployment strategy to this pipeline?"

Day 9 · Monitoring
"We Found Out from a Tweet"
Production was down for 2 hours. The team slept through it. A customer tweeted at 5:30am and that's how Minh found out. This can never happen again.
Prometheus Grafana Alertmanager 🔴 Silent outage
Locked ▾
📖 The Situation
At 3:17am, the database connection pool exhausted. Every API request started returning 500. Nobody knew. At 5:30am, a customer tweeted: "hey @ecotrack is your dashboard broken? getting 500 errors for 2+ hours lol." Minh saw it at 6am and woke you up.
M
Minh (CTO) · 6:04 AM
We were down for 2+ hours. Found out from Twitter. I'm losing my mind. Can you please set up something so we know BEFORE our customers do?
🎯 Your Mission
Deploy the kube-prometheus-stack Helm chart to your cluster. Verify Prometheus is scraping all pods and Grafana is accessible.
Build a Grafana dashboard with 5 panels: HTTP request rate, error rate (5xx/total), p99 latency, pod CPU/memory, and pod restart count. These are the 5 things you check first in any incident.
Configure 4 alerts in Alertmanager: (1) error rate > 5% for 2 min, (2) p99 latency > 2s for 5 min, (3) any pod restarting > 3 times in 10 min, (4) DB connection pool > 80% saturation.
Simulate the incident: deliberately exhaust the DB connection pool (open many connections without closing them). Verify your alert fires within 2 minutes. You must receive the notification before manually checking anything.
Write a 1-page runbook: "What to do when the DB connection alert fires." Include: how to verify, how to find the leaking service, how to temporarily increase the pool, and how to fix properly.
✅ This chapter is done when
Recreate the 3am incident. This time your alert fires at T+90 seconds. You get a notification. You open the Grafana dashboard and can immediately see which panel shows the problem. The runbook tells you exactly what to do next. Minh never finds out from Twitter again.

Day 10 · Capstone
"Fix It or Lose the Customer"
Your biggest customer ($200K ARR) says the dashboard takes 30 seconds to load. They're evaluating alternatives. You have 2 hours. Everything you've built leads here.
All tools Incident response 🔴 Customer escalation
Locked ▾
📖 The Situation
Meridian Corp — 500 users, $200K ARR, 18% of EcoTrack's total revenue — emails at 9am. Their account manager calls Minh immediately.
M
Minh (CTO) · 9:11 AM
Meridian's CTO just called me. Their emissions dashboard is taking 30+ seconds to load. They've been patient for 3 days but they're now formally evaluating Persefoni as an alternative. We need this fixed today, ideally in the next 2 hours before their 11am board meeting where they're presenting the data. This is all hands on deck.

You open your Grafana dashboard. The graphs tell you something is wrong but not exactly what. Time to investigate.
🎯 Your Mission
Diagnose first, fix second. Open Grafana. What does p99 latency show? When did it start? Correlate with the deployment history, recent commits, and traffic patterns. Form a hypothesis before touching anything.
Check the application logs for slow queries, N+1 database calls, or missing indexes. The emissions dashboard query likely scans a large table without an index — find the slow query in the logs.
Investigate Kubernetes resource limits: are pods being CPU throttled? Is a pod restarting under load? Are there enough replicas? Check kubectl top pods and your Grafana panels.
Fix the root cause. Deploy the fix through the pipeline — not manually. The CI/CD system you built in Day 8 is how you ship, even under pressure.
Write a 1-page incident report for Minh and the customer: timeline, root cause, fix applied, and what you're doing to prevent it recurring. This is what separates a DevOps engineer from a firefighter.
✅ This chapter is done when
Dashboard loads in under 1 second. Fix was deployed through the pipeline, not manually ✓ · Grafana shows latency back to normal ✓ · You can present your Grafana dashboard to your trainer and explain the timeline of what happened ✓ · Incident report written ✓ · Meridian stays a customer ✓
